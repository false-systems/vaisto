# vaisto_bpf — Design Document

## Overview

`vaisto_bpf` is a **separate Hex package** that compiles a restricted subset of Vaisto to eBPF bytecode. It is to Vaisto what Aya is to Rust — an independent library that targets eBPF, not a modification to the core language.

**This is not part of Vaisto.** It is a standalone package with its own repo, its own `mix.exs`, its own release cycle. It depends on `vaisto` as a library dependency. Vaisto core must never contain a single line of eBPF-related code.

This design doc lives in the Vaisto repo temporarily as a reference. The implementation lives elsewhere.

Vaisto core knows nothing about eBPF. The typed AST is the contract between them.

## Architecture

```
                    vaisto (core)                          vaisto_bpf (package)
                    ─────────────                          ────────────────────
  .va source  →     Parser
              →     TypeChecker    ─── typed AST ───→      BpfValidator
              →     CoreEmitter   (BEAM backend)           BpfEmitter → eBPF bytecode → ELF .o
              →     Emitter       (Elixir backend)         BpfLoader (Rust CLI)
                                                           BpfDecoder (auto-generated BEAM decoders)
```

### Dependency Direction

`vaisto_bpf` depends on `vaisto`. Never the other way around.

## Language Subset

eBPF programs are written in standard Vaisto with a module-level target declaration:

```scheme
(ns MyProbe :target bpf)
```

The entire module is eBPF-targeted. No mixing BEAM and eBPF functions in the same file.

### What's Allowed

- Fixed-width integer types: `:u8`, `:u16`, `:u32`, `:u64`, `:i8`, `:i16`, `:i32`, `:i64`
- Float types: `:f64`
- Boolean: `:bool`
- Record types (compiled to C-aligned structs)
- `defn` functions (compiled to eBPF functions)
- `if`/`match` expressions
- Bounded loops (with explicit upper bound)
- BPF map operations (`bpf/map-lookup`, `bpf/map-update!`, `bpf/map-delete!`)
- BPF helper calls (`bpf/get-current-pid-tgid`, `bpf/ktime-get-ns`, etc.)
- Arithmetic and comparison operators

### What's Rejected

- Closures / anonymous functions
- Recursion (direct or indirect)
- Unbounded loops
- Dynamic lists
- Strings (use fixed-size byte arrays)
- Arbitrary allocation (512-byte stack limit)
- BEAM-specific constructs (processes, spawn, send, receive)

The `BpfValidator` rejects these at compile time with clear error messages before any bytecode is generated.

## Type System

### Fixed-Width Types

In `:target bpf` modules, types map to concrete sizes:

| Vaisto type | eBPF representation | Size (bytes) |
|-------------|---------------------|--------------|
| `:u8`       | `u8`                | 1            |
| `:u16`      | `u16`               | 2            |
| `:u32`      | `u32`               | 4            |
| `:u64`      | `u64`               | 8            |
| `:i8`       | `s8`                | 1            |
| `:i16`      | `s16`               | 2            |
| `:i32`      | `s32`               | 4            |
| `:i64`      | `s64`               | 8            |
| `:f64`      | `double`            | 8            |
| `:bool`     | `u8`                | 1            |

Standard Vaisto `:int` is rejected in BPF modules — it's arbitrary precision on BEAM and has no fixed eBPF representation. You must be explicit about width.

### Struct Layout — C-Aligned

Record types in BPF modules follow C struct alignment rules:

- Each field is aligned to its own size (u64 → 8-byte boundary, u32 → 4-byte, etc.)
- Struct total size is rounded up to the largest field's alignment
- Padding is inserted automatically and invisibly

Example:

```scheme
(ns MyProbe :target bpf)
(deftype Event [flags :u8 pid :u64 count :u32])
```

Layout:

```
offset 0:   flags   (u8)    — 1 byte
offset 1:   [pad]           — 7 bytes (align pid to 8)
offset 8:   pid     (u64)   — 8 bytes
offset 16:  count   (u32)   — 4 bytes
offset 20:  [pad]           — 4 bytes (round struct to 8)
total: 24 bytes
```

**Rationale**: The entire eBPF ecosystem (kernel structs, BTF, bpftool, perf, every other tool) uses C struct layout. Going packed would make `vaisto_bpf` incompatible with everything. C-aligned is the only viable choice.

### Auto-Generated BEAM Decoders

For every BPF record type, `vaisto_bpf` generates a BEAM-side decoder module:

```elixir
# Auto-generated by vaisto_bpf — do not edit
defmodule MyProbe.Decoder do
  def decode_event(<<
    flags::native-unsigned-integer-8,
    _pad1::binary-size(7),
    pid::native-unsigned-integer-64,
    count::native-unsigned-integer-32,
    _pad2::binary-size(4)
  >>) do
    {:Event, flags, pid, count}
  end

  def encode_event({:Event, flags, pid, count}) do
    <<
      flags::native-unsigned-integer-8,
      0::binary-size(7),
      pid::native-unsigned-integer-64,
      count::native-unsigned-integer-32,
      0::binary-size(4)
    >>
  end
end
```

One type definition → two correct representations. The types are the contract between kernel and userspace.

## Bytecode Generation

### Direct Emission — No Clang

`vaisto_bpf` emits eBPF bytecode directly from Elixir. No clang, no LLVM dependency.

eBPF instructions are simple — each is 8 bytes:

```
| opcode (1 byte) | dst_reg:src_reg (1 byte) | offset (2 bytes) | immediate (4 bytes) |
```

~100 opcodes, 11 registers (r0-r10), register machine. The restricted Vaisto subset maps almost 1:1 to eBPF instructions.

The emitter produces an ELF `.o` file containing:
- `.text` section: eBPF instructions
- `.maps` section: map definitions
- `.rodata` section: read-only data (if any)
- Symbol table and relocations

### Compilation Pipeline

```
Vaisto AST → BpfValidator → BpfIR (linear instructions) → BpfAssembler → ELF .o
```

The `BpfIR` is an intermediate representation of linear eBPF instructions with symbolic register names, resolved to physical registers in the assembler pass.

## Program Loading

### Rust CLI Tool: `vaisto-bpf-loader`

A small Rust binary, bundled with the package. Called via `System.cmd` — not a port, not a NIF. A build/load tool, like how Mix calls `gcc`.

```bash
# Load an eBPF program and pin to bpffs
vaisto-bpf-loader load program.o --pin /sys/fs/bpf/my_probe

# Attach to a hook point
vaisto-bpf-loader attach /sys/fs/bpf/my_probe --tracepoint syscalls/sys_enter_read

# Detach and unpin
vaisto-bpf-loader detach /sys/fs/bpf/my_probe
vaisto-bpf-loader unpin /sys/fs/bpf/my_probe
```

Built with Aya (Rust eBPF library). Compiled during `mix compile` via Rustler-style build step.

## Runtime Data Flow

### Polling (Map Lookups)

For periodic reads (counters, histograms, aggregations):

1. eBPF program writes to maps in kernel
2. Maps are pinned to `/sys/fs/bpf/`
3. BEAM calls `vaisto-bpf-loader dump-map <path>` via `System.cmd`
4. Raw binary returned → decoded by auto-generated BEAM decoder
5. GenServer wraps this as a periodic poll

### Streaming (Ring Buffer)

For high-frequency event streams:

1. eBPF program pushes events to `BPF_MAP_TYPE_RINGBUF`
2. `vaisto-bpf-reader` (Rust daemon) reads ringbuf via `mmap` + `epoll`
3. Writes framed binary events to a Unix socket
4. BEAM GenServer connects via `:gen_tcp` on the Unix socket
5. Incoming binary frames decoded by auto-generated decoder
6. Events delivered to Vaisto processes

This is not a port — it's a dedicated runtime component with a proper binary protocol, like connecting to Redis or Postgres.

## Verifier Strategy

### Pre-Validation in Vaisto

The `BpfValidator` catches what it can at compile time, before any bytecode is generated:

- Recursion (direct or indirect) → reject
- Unbounded loops → reject
- Stack overflow (>512 bytes of local variables) → reject
- Unsupported types (closures, dynamic lists, strings) → reject
- Unchecked map lookup results → reject (must pattern match on nil)
- Uninitialized variables → reject

### Kernel Verifier Error Translation

For errors that only the kernel verifier can catch:

1. The verifier returns errors with eBPF instruction numbers
2. `vaisto_bpf` maps instruction numbers back to Vaisto source locations (via a source map generated during emission)
3. Errors are displayed in Vaisto's error format with source context:

```
error[BPF]: verifier rejected — possible null pointer dereference
  ┌─ probes/syscall_counter.va:12:5
  │
12 │   (bpf/map-lookup counter (get ctx :pid))
  │    ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  │
  = note: map lookups may return null — use match to handle both cases
```

## Supported Program Types

### v1: Tracepoints

First target. Stable interface, well-defined context structs, immediately useful for observability, simplest verifier requirements.

### Future

- `kprobe` / `kretprobe` — kernel function tracing
- `fentry` / `fexit` — modern BTF-based tracing
- `uprobe` / `uretprobe` — userspace function tracing
- `XDP` — high-performance packet processing
- `tc` — traffic control

## BTF/CO-RE

Skipped for v1. Programs target the kernel they compile on.

CO-RE (Compile Once Run Everywhere) will be added later for cross-kernel portability. This requires generating BTF type information and using relocations for kernel struct field access.

## Testing Strategy

| Layer | Environment | What's Tested |
|-------|-------------|---------------|
| BpfValidator | Any OS | Subset restriction enforcement |
| BpfEmitter | Any OS | AST → correct eBPF instruction sequences |
| BpfAssembler | Any OS | Instructions → valid ELF .o files |
| Decoder generation | Any OS | Type → correct binary pattern matching |
| Verifier acceptance | Linux CI | Generated programs pass kernel verifier |
| Integration | Linux + privileged | Full pipeline: compile → load → attach → read |

80% of the package is testable on macOS. Only loading/attaching needs Linux.

## Example: End-to-End

### BPF module (`probes/syscall_counter.va`)

```scheme
(ns SyscallCounter :target bpf)

(deftype Key [pid :u32])
(deftype Value [count :u64])

(extern bpf/get-current-pid-tgid [] :u64)
(extern bpf/map-lookup [counter Key] (Option Value))
(extern bpf/map-update! [counter Key Value] :u32)

(defn probe [ctx :bpf/tracepoint-context] :u32
  (let [pid (u32 (>> (bpf/get-current-pid-tgid) 32))
        key (Key pid)]
    (match (bpf/map-lookup counter key)
      [(Some val) (bpf/map-update! counter key (Value (+ (. val count) 1)))]
      [None       (bpf/map-update! counter key (Value 1))])))
```

### BEAM-side orchestration (`lib/my_app/tracing.va`)

```scheme
(ns MyApp.Tracing)
(import SyscallCounter.Decoder)

(process syscall-monitor {:last-counts {}}
  :poll
    (let [raw   (bpf/dump-map "/sys/fs/bpf/syscall_counter")
          counts (SyscallCounter.Decoder/decode-all raw)]
      counts)
  :get state)
```

### Workflow

```bash
# Compile BPF module
vaistoc probes/syscall_counter.va --target bpf -o build/syscall_counter.o

# Load and attach
vaisto-bpf-loader load build/syscall_counter.o --pin /sys/fs/bpf/syscall_counter
vaisto-bpf-loader attach /sys/fs/bpf/syscall_counter --tracepoint syscalls/sys_enter_read

# Run BEAM application that reads the maps
vaistoc run lib/my_app/tracing.va
```

## Open Questions for Later

- **CO-RE / BTF support** — needed for cross-kernel portability
- **Ring buffer streaming protocol** — framing format for the Unix socket
- **Map type system** — how to declare and type-check BPF map definitions in Vaisto
- **Tail calls** — eBPF supports tail calls between programs (up to 33 deep)
- **Per-CPU maps** — important for high-frequency counters, needs API design
- **bpffs management** — pinning, unpinning, namespace isolation
